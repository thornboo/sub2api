# dev-zz 分支策略

`dev-zz` 是在上游 `main` 基础上维护的二开分支。它需要持续吸收上游修复，同时保留本分支已经形成的产品、视觉、部署和运维策略。

## 基本原则

- 上游的正确性修复、兼容性修复、安全修复和后端运行时修复优先吸收。
- dev-zz 已明确保留的行为，不因上游合并自动丢弃。
- 处理冲突时，先确认 dev-zz 的现有策略，再判断上游变更是补充、替代还是冲突。
- 每次合并完成后，必须记录到 [上游合并记录](./maintenance/merge-log.md)。

## 默认保留的 dev-zz 策略

- 保留 dev-zz 的前端视觉方向和控制台样式，除非明确决定跟随上游重做。
- 保留前端隐藏特定第三方登录入口的策略，后端能力不因此删除。
- 保留永久数据保留的默认策略，除非有新的运维决策。
- 保留账号模型探测、模型映射和 models.dev 查询等二开能力。
- 保留 dev-zz 从源码构建部署的策略，不直接依赖上游公开镜像。

## 合并判断

上游变更通常分三类处理：

| 类型 | 处理方式 |
| --- | --- |
| 后端正确性修复 | 优先接受，除非破坏 dev-zz 已记录的策略 |
| 前端 UI / 交互变更 | 对照 dev-zz 的视觉和体验策略再合并 |
| 部署 / 文档变更 | 保留 dev-zz 的源码构建部署策略，同时吸收通用修复 |

## 记录要求

- 合并 main：更新 [上游合并记录](./maintenance/merge-log.md)。
- 用户可见变化：更新 [变更记录](./changelog.md)。
- 补丁实现：更新 [补丁记录](./patches.md)。
- 长期设计决策：放入 `docs-site/dev-zz/decisions/`。
