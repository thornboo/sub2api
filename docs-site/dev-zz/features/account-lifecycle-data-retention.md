# 上游账号生命周期与数据保留策略

> 状态：方案稿。用于定义账号停用、归档和永久删除的产品语义；尚未落地实现。

## 背景

当前管理端账号删除路径更接近软删除：账号从常规管理视图消失，但历史 `usage_logs` 仍然保留，历史用量、费用和请求数不会因为普通删除而自动扣减。

这符合 dev-zz 的数据保留原则：usage 是消费账本，不是普通运行日志。但现状有两个问题：

1. 管理员看到的是“删除账号”，语义上容易误解为历史记录也被清理。
2. 账号软删除后，历史 usage 记录仍在，但账号名称等上下文可能解析不到，历史账本可读性下降。

这页把账号生命周期拆成三个动作：

- 停用：停止新调度和新请求使用。
- 归档：从日常管理视图隐藏，但保留历史账本和审计上下文。
- 永久删除：显式清除该账号相关业务记录，并承认历史统计会变化。

## 目标

- 保留业务事实数据，特别是 usage、计费、审计和争议处理需要的证据。
- 允许管理员把不用的账号从日常管理视图中收起。
- 支持受限的永久删除能力，用于清理测试账号、误导入账号或合规要求下的彻底清除。
- 明确永久删除会改变历史统计，不把它伪装成普通删除。
- 继续允许垃圾日志、运行噪声和过期派生缓存按显式清理策略删除。

## 非目标

- 不把所有日志都永久保留。调试日志、ops 噪声、临时缓存和可重建派生数据仍可清理。
- 不把账号归档等同于 usage 清理。
- 不让普通账号管理列表混入已归档账号，除非管理员显式切换视图。
- 不在第一阶段改写计费算法、网关转发、余额扣费或 usage 写入口径。

## 数据分类

| 类型 | 示例 | 默认策略 |
| --- | --- | --- |
| 业务事实账本 | `usage_logs`、扣费事实、请求成本、计费倍率快照 | 默认保留，不随账号归档删除 |
| 当前配置实体 | `accounts`、`account_groups`、账号凭据、账号状态、调度参数 | 可停用、归档、恢复 |
| 历史解释快照 | usage 写入时的账号名、平台、分组、Key 名称 | 应随账本保留 |
| 显式审计事件 | 停用、归档、恢复、永久删除的操作者和结果 | 应保留最小必要信息 |
| 派生统计与缓存 | dashboard 聚合、查询缓存、scheduler snapshot | 可重算；永久删除后必须失效或重建 |
| 垃圾日志 / 运行噪声 | debug 日志、过期 ops 明细、临时任务输出 | 可按显式清理策略删除 |

## 生命周期状态机

推荐产品状态：

```text
正常 / 启用
  -> 停用
  -> 归档
  -> 永久删除
```

不可跳级：

- 正常账号不能直接归档。
- 正常或停用账号不能直接永久删除。
- 永久删除只能从已归档账号进入。

### 正常 / 启用

语义：

- 账号可以参与网关请求转发、模型调度、自动探测和自检。
- 账号出现在常规账号列表、筛选器和调度候选中。
- 新请求可以继续写入该账号的 usage。

实现注意：

- 当前系统已有 `status`、`schedulable`、限流和临时不可调度等概念。生命周期实现前，需要决定“正常”是否继续复用现有 `status = active` 和 `schedulable = true`。

### 停用

语义：

- 停止该账号参与新请求、新调度、自动探测和自检。
- 历史 usage、账号配置、分组关系和凭据仍保留。
- 账号仍在常规账号管理页显示，管理员可以恢复、编辑或进入归档。

前置条件：

- 账号存在且未归档。

数据行为：

- 不删除 `usage_logs`。
- 不删除 `account_groups`。
- 不删除 scheduled test plans；可以自动暂停其执行，但不要清除配置。
- 调度缓存和 scheduler snapshot 必须及时失效。
- 停用不得复用当前账号删除路径；删除路径会清理分组关系和 scheduled test plans，不符合停用语义。

UI 行为：

- 主列表显示为“已停用”。
- 主操作是“恢复启用”和“归档账号”。
- 停用确认文案说明：历史用量和统计保留。

### 归档

语义：

- 账号不再出现在默认账号管理列表、账号筛选下拉、调度候选、探测候选和自检候选中。
- 历史 usage、统计、审计和账号解释信息保留。
- 管理员可以在“已归档账号”视图中查看、恢复或发起永久删除。

前置条件：

- 账号必须已停用。
- 账号不能有正在运行的探测、刷新、导入或其它账号级后台任务。

数据行为：

- 不删除 `usage_logs`。
- 不删除历史 usage 相关的账号名称展示能力。
- 不删除账号本体的最小可读信息。
- 可以停止或暂停后续 scheduled test plans，但第一阶段不应硬删计划。
- 归档不得复用当前 `accountRepository.Delete` 路径；该路径会删除账号分组关系和 scheduled test plans，不符合归档保留上下文的语义。

实现选型：

| 方案 | 优点 | 风险 |
| --- | --- | --- |
| 复用当前 soft delete 的 `deleted_at` | 改动小，默认查询天然隐藏归档账号 | 历史 usage hydrate 需要显式穿透软删除；“归档”和“删除”命名要整体改掉 |
| 新增 `archived_at` | 语义清楚，可与删除区分 | 需要改账号列表、调度查询、筛选器和索引 |
| 新增 `lifecycle_state` | 状态机清晰，便于表达 `active/disabled/archived/purged` | 迁移和兼容成本较高，要和现有 `status` 拆分职责 |

推荐先评估 `archived_at` 或 `lifecycle_state`，不要继续让“删除”承担归档语义。若短期为了降低成本复用 `deleted_at`，UI 和接口也应改名为“归档”，并把永久删除设计成单独 Purge 流程。

UI 行为：

- 默认列表不显示已归档账号。
- 账号管理页增加“已归档”视图或筛选。
- 已归档账号详情只能查看历史上下文、恢复或永久删除，不允许直接编辑凭据并重新参与调度。
- 从已归档恢复时，恢复到“停用”状态，而不是直接恢复为“正常启用”。

### 永久删除 / Purge

语义：

- 明确删除该账号相关业务记录。
- 不可恢复。
- 会改变历史 usage 统计、账号维度费用、模型分布、错误明细和任何依赖该账号 usage 的历史报表。

前置条件：

- 账号必须已归档。
- 必须完成影响预估。
- 必须有高权限角色确认，建议只允许 owner / super admin。
- 必须二次确认，建议输入账号名称或账号 ID。
- 生产环境建议设置冷却期，例如归档后至少等待 24 小时才能永久删除。

Purge 不应只是 hard delete `accounts` 一行。它应该是专门的数据清除任务，至少处理：

- `usage_logs.account_id = <account_id>` 的历史记录。
- 账号分组关系。
- scheduled test plans 和历史探测结果。
- model self-check histories。
- ops / request details 中指向该账号的业务排障记录。
- scheduler outbox、调度快照和账号缓存。
- 以 `account_ids` 数组引用该账号的配置，例如账号统计定价规则。
- dashboard 聚合、查询缓存和其它可重建派生统计。

如果某些表只属于垃圾日志或可清理运行噪声，可以删除；如果某些表属于合规审计，必须先决定保留最小审计还是一起清除。

Purge 的 usage 处理必须在设计阶段先做选择。当前 `usage_logs.account_id` 是必填账号维度；“匿名化”不能简单把 `account_id` 置空，必须选择删除 usage 行、引入匿名 / 墓碑账号、调整 schema 为可空，或迁移到脱敏证据表。

## 统计口径

### 归档后的统计

归档后：

- 全局 usage 请求数、token、费用不变。
- 按旧 `account_id` 查询历史 usage 仍应返回数据。
- admin usage 列表应显示账号名，并标记“已归档”。
- dashboard 的“当前账号数量 / 正常账号 / 异常账号”会减少或排除归档账号。
- 账号筛选器默认不显示已归档账号；历史证据查询可以提供“包含已归档账号”开关。

这是一组刻意设计的差异：

- 历史消费：保留。
- 当前资源：隐藏或排除。

### 永久删除后的统计

永久删除后：

- 该账号关联的 usage 明细会被删除或匿名化。
- 全局 usage 请求数、token、费用会下降。
- 按账号维度的历史统计会消失。
- dashboard 聚合和缓存必须失效或重算。
- 导出和历史报表应能解释“该时间段曾执行 Purge，报表口径已改变”。
- 默认不自动退款、不自动回滚用户余额或订阅用量；如果业务需要财务冲正，必须走单独的余额 / 订阅调整审计流程。

Purge 前的确认文案必须直接说明：

```text
永久删除会删除该账号相关历史用量和统计数据，历史报表将发生变化。此操作不可恢复。
```

## 审计要求

建议新增账号生命周期审计事件，记录：

- 操作类型：`disable`、`enable`、`archive`、`restore`、`purge_requested`、`purge_completed`、`purge_failed`。
- 操作者 ID。
- 目标账号 ID。
- 操作时间。
- 操作原因，可选但推荐。
- 影响预估和最终影响结果，例如删除 usage 行数、清理表数量、失效缓存范围。

Purge 的审计要特别处理敏感信息：

- 不保留账号凭据、access token、refresh token、API key、cookie。
- 可以保留账号 ID、删除时间、操作者和影响统计。
- 如果合规要求“完全不留账号身份”，则审计只保留不可逆摘要和影响数量，但这会降低排障能力。

## API 设计草案

### 停用账号

```http
POST /api/v1/admin/accounts/:id/disable
```

行为：

- 只允许未归档账号。
- 停止调度和探测。
- 保留所有历史数据。
- 写入 lifecycle audit。

### 恢复启用

```http
POST /api/v1/admin/accounts/:id/enable
```

行为：

- 只允许停用账号。
- 不允许已归档账号直接启用；已归档账号必须先 restore 到停用。

### 归档账号

```http
POST /api/v1/admin/accounts/:id/archive
```

行为：

- 只允许停用账号。
- 归档后从默认账号列表隐藏。
- 保留 usage 和历史证据。
- 写入 lifecycle audit。

### 恢复归档账号

```http
POST /api/v1/admin/accounts/:id/restore
```

行为：

- 只允许已归档账号。
- 恢复到停用状态。
- 管理员再显式启用，账号才重新进入调度。

### Purge 影响预估

```http
GET /api/v1/admin/accounts/:id/purge-preview
```

响应至少包含：

```json
{
  "account_id": 123,
  "account_name": "OpenAI Main",
  "usage_logs": 42015,
  "estimated_total_cost": 91.23,
  "scheduled_test_plans": 3,
  "model_self_check_histories": 220,
  "ops_records": 19,
  "affected_aggregates": ["dashboard_usage", "account_usage"],
  "will_change_historical_reports": true
}
```

### 发起 Purge

```http
POST /api/v1/admin/accounts/:id/purge
```

请求：

```json
{
  "confirmation": "OpenAI Main",
  "reason": "测试账号误导入，确认清理历史用量",
  "preview_token": "..."
}
```

行为：

- 校验账号仍处于归档状态。
- 校验 preview token 未过期，且账号影响范围未发生重大变化。
- 创建异步 purge task。
- 返回任务 ID。

Purge 任务完成后，必须返回最终影响数量，并写审计事件。

## UI 设计草案

账号管理默认操作：

| 当前状态 | 主操作 | 危险操作 |
| --- | --- | --- |
| 正常 | 停用 | 无 |
| 停用 | 恢复启用、归档 | 无 |
| 归档 | 恢复到停用 | 永久删除 |

交互要求：

- 默认“删除”按钮改名为“归档账号”。
- 只有已归档账号详情中显示“永久删除”。
- 永久删除用独立危险弹窗，不和归档共用确认框。
- Purge 弹窗先显示影响预估，再要求输入账号名称或 ID。
- Purge 文案必须说明历史统计会变化。
- 已归档账号在 usage 明细中显示 `账号名（已归档）`，而不是 `-`。
- 历史证据筛选器可提供“包含已归档账号”开关。

## 实施阶段

### 阶段 1：修正归档语义

目标：

- 把账号删除产品语义改成归档。
- 停用后才能归档。
- 已归档账号可恢复到停用。
- admin usage 能显示已归档账号名称和状态。
- 历史 usage 统计不变。

建议范围：

- 后端新增 lifecycle service 或在 account service 中集中处理状态迁移。
- 前端账号管理按钮和确认文案改名。
- admin usage hydrate 账号时支持解析已归档账号。
- 账号列表增加“已归档”视图。

### 阶段 2：历史快照

目标：

- usage 写入时保存账号名称、平台和必要展示快照。
- 账号改名、归档或恢复后，历史 usage 展示不漂移。
- 导出使用快照名称。

这部分应和 [用量账本与已删除 Key 证据完整性](./usage-ledger-evidence-integrity.md) 的阶段 2 合并考虑，避免重复加列或重复定义快照优先级。

### 阶段 3：Purge 流程

目标：

- 已归档账号支持永久删除。
- Purge 先预估影响，再异步执行。
- Purge 完成后清理 usage、派生统计、缓存和关联记录。
- 历史报表变化可解释。

实现前置：

- 列出所有直接或间接引用 `account_id` 的表和配置。
- 查清生产库真实外键约束，尤其是 `usage_logs.account_id` 是否仍存在级联删除历史。
- 查清 `usage_logs.account_id` 的必填约束和匿名化方案；当前 schema 下不能直接把账号维度置空。
- 明确哪些表属于业务事实、哪些属于垃圾日志、哪些属于可重建派生数据。
- 明确 Purge 是否只改变历史报表，还是同时触发余额 / 订阅用量冲正；默认不冲正。
- 为 purge task 增加幂等键和失败恢复策略。

## 验收用例

### 归档语义

1. 正常账号不能直接归档。
2. 停用账号可以归档。
3. 归档账号不会出现在默认账号列表。
4. 归档账号不会进入调度候选。
5. 归档账号不会被自动探测或模型自检选中。
6. 归档账号的历史 usage 明细仍可查询。
7. 归档账号的历史 usage 统计不变。
8. admin usage 明细显示 `账号名（已归档）`。
9. 归档账号恢复后处于停用状态，不会直接参与调度。

### Purge 语义

1. 未归档账号不能 purge。
2. Purge preview 返回 usage 行数、影响统计和风险提示。
3. confirmation 不匹配时拒绝 purge。
4. Purge 删除目标账号的 usage 后，全局 usage 统计按预期下降。
5. Purge 不影响其它账号的 usage。
6. Purge 完成后账号无法恢复。
7. Purge 完成后相关缓存失效或重算。
8. Purge 写入最小审计记录。
9. Purge 失败可重试，不产生重复清理或半成功误报。
10. Purge 完成后默认不触发余额退款或订阅用量冲正；需要财务冲正时必须走单独审计流程。

## 开放问题

1. 归档实现采用 `archived_at`、`lifecycle_state`，还是短期复用 `deleted_at`？
2. 停用语义最终映射到 `status = disabled`、`status = inactive`，还是 `schedulable = false`？
3. scheduled test plans 在归档时是暂停、保留并隐藏，还是移动到归档账号详情中只读展示？
4. Purge 是否要删除 ops 业务排障记录，还是只断开账号维度并保留匿名错误证据？
5. Purge 审计是否保留账号名称，还是只保留账号 ID / 不可逆摘要？
6. 历史聚合表是 purge 时局部重算，还是直接失效并由后台重新聚合？
7. 生产环境是否强制归档冷却期？冷却期多长？

## 当前建议

第一期先做归档语义，不做 Purge：

1. 把账号管理中的“删除”改成“归档”。
2. 要求停用后才能归档。
3. 归档保留历史 usage 和统计。
4. admin usage 能显示已归档账号名称。
5. 已归档账号可以恢复到停用。

第二期再做 Purge：

1. 只允许已归档账号发起。
2. 先展示影响预估。
3. 使用异步任务执行。
4. 清理 usage 和派生统计。
5. 明确历史报表会变化。
