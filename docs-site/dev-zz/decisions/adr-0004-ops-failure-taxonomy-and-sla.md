# 设计取舍 0004：拆分客户可见失败、失败归因与平台 SLA

## 结论

已采纳。

日期：2026-07-18。

本 ADR 取代“使用 `is_business_limited` 一个布尔值同时决定客户侧限制、错误归因和 SLA 归属”的临时口径。旧字段在迁移期保留兼容，但新的运维看板、趋势、告警、健康评分、定时报表和错误明细必须逐步切换到结构化失败分类。

完整数据合同、分类矩阵、指标公式、页面结构、迁移顺序和验收样本见 [运维失败归因、SLA 与事件视图重构](../features/ops-failure-classification-redesign.md)。

## 背景

第一版运维排障把失败拆成：

- 客户可见失败；
- 客户侧限制；
- SLA 错误；
- 上游限流或过载。

该版本解决了“客户报错但管理员只看 SLA 错误”的入口问题，但仍以 `is_business_limited` 作为核心二分条件：

```text
is_business_limited = true  -> 客户侧限制，不计入 SLA
is_business_limited = false -> SLA 错误
```

2026-07-17 的线上六小时窗口证明这个二分法不能表达真实责任：

- 4,813 条平台托管 OpenAI 分组“无可用上游账号”被归入“客户侧限制”；
- 72 条企业成员预算耗尽被归入“SLA 错误”；
- 312 条所谓 SLA 错误中至少 208 条属于账户、权限、预算、模型或请求兼容性；
- 两次主要故障被 Codex 客户端重试放大成近一万条失败请求，页面无法表达“历史窗口很差但当前已经恢复”。

总数大体正确，但责任归因、SLA 和运维动作被混在了一起。

## 决策

### 1. 三个问题必须正交表达

每个错误事件分别回答：

1. 客户是否最终看到了失败；
2. 失败发生在哪个领域、需要谁处理；
3. 是否计入平台对客户的 SLA。

任何一个字段都不能再由另一个字段隐式代替。

### 2. 采用结构化失败分类

新分类至少包含：

- `event_scope`：终态请求、流式终态失败或已恢复的上游尝试；
- `customer_visible`：客户最终是否看到失败；
- `failure_domain`：customer / enterprise / client / platform / upstream / unknown；
- `failure_category`：余额、预算、权限、协议、路由容量、上游限流、超时、内部依赖等稳定类别；
- `failure_reason`：可过滤的闭集原因码；
- `resolution_owner`：customer / enterprise_admin / platform_ops / client / unknown；
- `pool_ownership`：platform / enterprise / unknown；
- `sla_impact`：true / false / unknown；
- `classification_version`：记录写入时使用的分类规则版本。

已有 `error_phase`、`error_owner`、`error_source` 继续表达技术发生位置；新字段表达产品归因与运维动作，两者不能互相覆盖。

### 3. 平台托管账号池不可用计入平台 SLA

当请求已经通过认证、余额和权限检查，所请求模型也属于平台提供范围，但平台托管账号池因下列原因无法选出账号时，应计入平台 SLA：

- 账号池为空；
- 所有账号不可调度；
- 所有账号处于 cooldown、runtime block 或自动暂停；
- 平台维护的账号余额、凭据或上游状态导致池归零；
- 分组内没有有效冗余，唯一账号失败后无法继续服务。

客户不能修复平台维护的账号和调度状态，因此这类 503 不能显示成“客户侧限制”。

### 4. 企业自管账号池单独归因

未来若允许企业自行维护上游账号，同样的“无可用账号”应记录为企业自管上游问题，默认不计入平台 SLA。

因此 `pool_ownership` 必须在错误发生时快照化，不能在查询时根据当前配置反推历史责任。

当前产品的上游账号均由平台管理员维护，现有历史“无可用账号”可以按平台托管口径处理；引入企业自管账号前必须先完成 ownership 快照。

### 5. 最终上游失败计入平台 SLA，已恢复尝试不计入

- 客户最终收到的上游 429、5xx、超时、认证或上游余额失败，属于 `upstream` 领域，并计入平台对客户的 SLA；同时保留上游原因，方便平台向供应商追责。
- 某次上游尝试失败但 failover 后客户最终成功，只进入上游健康观察，不进入客户可见失败和 SLA 失败。

### 6. 客户、企业策略与客户端错误不计入平台 SLA

下列错误仍然可见、可筛选、可告警给对应责任方，但不计入平台 SLA：

- 用户余额不足；
- API Key 无效、停用、过期或额度耗尽；
- 企业成员预算耗尽；
- 企业分组、模型、白名单或功能策略限制；
- 请求体、协议或模型名称不合法；
- 客户端主动取消、断开或本地超时。

### 7. 请求次数与故障事件同时表达

页面保留原始“失败请求次数”，并明确注明包含客户端重试；后续增加基于稳定指纹和时间间隔的故障事件聚合。

事件聚合不能删除或合并底层错误证据，只是额外的运维视图。

### 8. 历史数据只做确定性迁移

- 能根据现有错误码、阶段、上游上下文和当前平台托管边界确定分类的记录可以回填；
- 无法可靠判断的历史记录标记为 `unknown`，不能为了让图表完整而猜测；
- 存在未知分类时，界面必须显示数据质量提示，不能把未知静默当成健康或客户责任。

## 必须成立的系统不变量

1. `customer_visible`、`failure_domain` 和 `sla_impact` 是三个独立字段。
2. 只有终态客户失败才进入客户可见失败率；已恢复的上游尝试不进入。
3. 只有 `sla_impact=true` 的终态失败进入平台 SLA 分子。
4. 客户余额、企业预算、权限和客户端取消不得进入平台 SLA。
5. 平台托管路由容量归零必须进入平台 SLA。
6. `pool_ownership`、分类版本和 SLA 归属是发生时快照，后续配置变化不得重写历史。
7. 原始聚合、预聚合、趋势、明细、导出、告警、健康评分和定时报表必须使用同一分类合同。
8. 卡片数量和使用同一时间窗口、同一筛选条件打开的明细数量必须一致。
9. 流式 HTTP 200 后终态失败不能同时被计为成功和失败。
10. 未分类记录必须单独计数，并触发数据质量观察，不能被默认归到客户侧或平台侧。

## 被拒绝的方案

### 只修改前端文案

拒绝。它不能修复平台容量错误进入客户侧、成员预算进入 SLA 的根本问题，也不能修复健康评分、告警和定时报表。

### 继续扩充 `is_business_limited`

拒绝。单个布尔值无法同时表达客户可见性、发生领域、处理方、上游池所有权和 SLA 归属。

### 仅按 HTTP 状态码归因

拒绝。同一个 403 可能是客户余额、企业预算、上游凭据或上游余额；同一个 503 可能是平台路由容量、上游过载或内部依赖故障。

### 将所有上游失败排除出平台 SLA

拒绝。客户购买的是平台服务，平台可以在内部保留供应商归因，但不能因为根因位于第三方就把最终客户失败从平台可用性中移除。

### 强行回填全部历史记录

拒绝。历史记录缺少 ownership 或明确原因时，强行归类会制造看似精确但不可审计的数据。

## 后果

- `ops_error_logs` 需要新增结构化分类与发生时快照字段。
- 错误写入路径需要从字符串猜测升级为“显式类型标记优先、历史文本回退”的分类器。
- dashboard raw、preagg、trend、distribution、detail、alert、health score 和 scheduled report 都要迁移。
- 前端卡片改为“客户可见失败 + 平台可用性 + 归因分布 + 当前状态”，不再展示二分的“客户侧限制 / SLA 错误”。
- 旧接口字段在迁移期保留，并标记为兼容字段；新代码不得继续以它们作为唯一判断依据。
- 需要用 2026-07-17 的真实六小时窗口建立固定回归样本，防止分类再次倒置。
