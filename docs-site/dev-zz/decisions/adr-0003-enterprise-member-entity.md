# 设计取舍 0003：以不可登录成员实体承载企业成员、聚合 Key 与成员预算

## 结论

已采纳。取代 [ADR-0002](./adr-0002-key-as-enterprise-member.md) 中“不引入成员实体、一把 Key = 一个员工席位”的临时取舍。

日期：2026-07-12；预算授权语义于 2026-07-20 修订。

本 ADR 记录的是完整目标架构，不是 MVP 或临时兼容方案。实现可以按依赖顺序拆分和验证，但不得以阶段性交付为理由缩减下述领域边界、权限约束、审计证据或一致性要求。

## 决策

### 1. 成员是独立但不可登录的领域实体

- 企业账号名下拥有多个 `enterprise_members`。
- 成员不能登录，不签发站点会话，不进入管理员授权角色体系。
- 成员可以拥有多把 API Key；每把成员 Key 通过成员继承有序分组集合。
- 成员是预算、用量归集、状态控制、导入迁移和审计追踪的稳定主体。

### 2. 企业能力不作为新的授权角色

`users.role` 继续只表达系统授权角色：`admin` / `user`。

新增独立账号类型 `users.account_type`：

- `individual`：普通用户。
- `enterprise`：企业账号，启用成员管理能力。

企业账号仍然是 `role=user`。这样把“是否为管理员”和“是否拥有企业成员能力”分成两个正交维度，避免把产品类型混入权限角色。

平台管理员必须能够创建企业账号或把尚未产生企业成员事实的普通账号升级为企业账号；管理员仍只把企业账号视为一个结算用户，不读取成员明细。企业账号一旦产生成员、成员 Key、成员预算或成员用量证据，不允许直接降回普通账号，只能停用企业能力并保留历史。

### 3. 企业余额、成员聚合限额和 Key 限额是三个不同层级

- 企业余额是平台结算的唯一资金池。
- 成员 5h、1d、7d 和自然月限额是企业内部的聚合控制边界，由成员名下全部 Key 共同消费。普通同步请求在执行前只检查当前实际已结算用量，并创建零金额持久化请求回执；实际费用结算达到或超过限额后阻止后续请求。
- 允许最后一笔同步请求或少量并发在途请求形成有限超额。这与企业余额、Key quota 和订阅额度的实际费用后结算语义保持一致，不能再用理论最大费用提前拒绝仍有实际余额的成员。
- Batch image、异步视频等在 HTTP 响应结束后仍会继续产生费用的任务，可以按明确预计任务费用创建持久化金额占用，并在任务完成、失败或结果不明时捕获、释放或进入核对。
- Key quota 与 Key 自身的 5h/1d/7d 限额是单 Key 控制，与成员聚合限额叠加生效，以最先耗尽的一层为准。
- 成员限额只累计用户实际应付金额，不使用管理员专属的上游成本或利润字段。
- 请求用量、迁移期开账和人工调整分别留下不可变账本证据；月度计数只是可重建投影，不是唯一事实来源。

### 4. 多分组路由是请求级编排，不是 API Key 字段扩容

成员 Key 鉴权后不立即绑定单个分组。网关必须先根据入口、请求模型、客户端能力和当前权限计算有序候选分组，再为每次尝试建立请求级 `ActiveGroup`：

1. 认证 Key、企业账号和成员。
2. 计算成员绑定与企业当前可用分组的交集。
3. 按入口能力和模型可调度性筛选候选分组。
4. 为当前候选创建不可变的请求级路由上下文。
5. 根据 `ActiveGroup.Platform` 分派到正确协议 handler。
6. 先完成分组内账号 failover；仅在响应尚未提交且错误可重试时，切换到下一个成员候选分组。

协议转换实现可以复用，但路由分派、请求上下文、错误分类、尝试记录和计费接入必须支持请求级分组，不能继续假设缓存 API Key 永远只有一个 `Group`。

### 5. 历史证据不可因配置变化而重写

- `usage_logs` 保存 `member_id`、请求发生时的成员名称快照和实际执行分组。
- 成员改名、换 Key、调整分组、停用或归档，不改变历史请求归属。
- 迁移期开账和人工预算调整写入成员预算账本，不伪造 usage log。
- 成员归档是可恢复的工作区状态；owner 的永久删除意图不能因为证据保留而变成“永远不可删除”的产品死端。
- 永久删除只允许从归档状态进入。服务端在事务内持锁复检：无历史事实时物理删除；存在历史事实时注销成员 Key、清除当前分组绑定并把成员转为不可见墓碑，保留用量、账本和审计关系，同时释放原成员编号供 owner 重新使用。

## 必须成立的系统不变量

1. 成员 Key 的 `user_id` 必须等于成员的 `enterprise_user_id`。
2. `member_id IS NOT NULL` 时，`group_id` 必须为空；普通 Key 可以继续使用单值 `group_id`。
3. 成员有效分组始终是“成员绑定 ∩ 企业账号当前可用分组 ∩ 当前入口/模型可调度分组”。
4. 企业账号分组授权被收回后，旧绑定和旧缓存都不能继续提供访问。
5. 成员禁用、归档或企业能力停用后，全部成员 Key 必须立即失效。
6. 归档恢复后成员保持 `disabled`，必须由 owner 核对分组并显式启用。
7. 成员路由不能通过分组自身的 fallback 配置逃出成员有效分组集合。
8. 一个逻辑请求无论经历多少账号或分组尝试，只能产生一次用户扣费和一次成员实际用量入账；失败尝试保留运维证据。
9. 成员 5h、1d、7d 和自然月限额的授权、回执/异步占用、实际结算与释放必须使用同一持久化证据链，具备幂等键和崩溃恢复能力；普通同步请求的回执金额必须为零。
10. 普通用户 Key（`member_id` 为空）的鉴权、路由、计费和用量行为保持兼容。

## 被拒绝的方案

### 继续使用“一把 Key = 一名员工”

拒绝。它不能表达一名成员多 Key、统一成员预算，也无法让单 Key 以确定顺序访问多个分组。

### 引入可登录子账号

拒绝。当前诉求不要求员工登录；为此改造认证、会话、支付、权限和账号生命周期会引入不必要的系统主体。

### 使用 `users.role = enterprise`

拒绝。`enterprise` 是产品能力和账号形态，不是管理员授权角色。把它放进 `role` 会扩大现有 admin/user 判断面，并使“企业账号仍是普通非管理员用户”的语义变得含混。

### 只在 Key 上增加多分组数组

拒绝。它仍然缺少稳定成员身份、成员级预算、多 Key 聚合、成员生命周期和迁移审计。

### 默认对同步请求采用理论最大费用预留

拒绝。理论上界受输出上限、模型映射、候选分组和倍率影响，会在成员仍有实际余额时提前拒绝请求，且与项目其他额度的实际费用后结算语义不一致。同步请求改为“当前实际用量预检 + 零金额持久化回执 + 实际费用原子结算”；接受最后一笔或少量并发请求的有限超额，并在实际用量达到限额后阻止后续请求。需要绝对硬上限时，必须另行设计成显式可选模式。

## 后果

- 新增企业账号类型、成员、成员分组、成员预算周期、请求回执/异步金额占用和预算账本数据结构。
- `api_keys` 增加可空 `member_id`；`usage_logs` 增加成员归属及快照字段。
- API Key 认证缓存升级为可表达成员与候选分组快照的版本化结构。
- 网关增加请求级路由编排层，并把实际执行分组传递给调度、计费、订阅、粘性会话、usage 和 ops。
- 企业用户端以成员管理替代裸 Key 管理；管理员端只增加企业账号类型的创建和状态管理，不增加成员明细入口。
- 导入采用服务器权威预览、确认令牌和事务内重新校验，支持 CSV 与受限 XLSX，不信任客户端解析结果。
