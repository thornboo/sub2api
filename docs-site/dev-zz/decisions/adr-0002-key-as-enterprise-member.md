# 设计取舍 0002：用 Key 承载企业成员管理，不引入子账号实体

## 结论

**已被 [ADR-0003](./adr-0003-enterprise-member-entity.md) 取代（2026-07-12）。**

本 ADR 的核心结论“不引入成员实体、一把 Key = 一个员工席位”是当时赶工落地的临时取舍，未解决一成员多 Key、聚合 Key、多分组路由、成员预算和迁移审计。ADR-0003 改用 `role=user, account_type=enterprise` 与**不可登录的成员实体**，并把请求级路由、预算预留/账本和历史证据纳入完整合同。它仍拒绝可登录子账号，但明确推翻“不引入成员实体”和“Key 即成员”的结论。

本 ADR 已落地的能力（批量创建、标签、批量管理、公共 Key 状态查询、owner 用量分析）对**普通用户 Key 保留有效**，与新方案并存。以下内容保留作为历史记录。

---

原结论：已采纳。

最近同步：2026-06-15。第一阶段批量发 Key、第二阶段标签和第三阶段 owner 用量分析第一版已经落地；公共 Key 状态查询也已落地，但仍不引入员工登录主体。

## 背景

部分企业客户在 sub2api 里只有一个普通账号（`users` 表一行），但需要给旗下多名员工分别发 Key、限制额度，并观察每个人的 API 用量。

参考项目 `claude-code-hub`（`ding113/claude-code-hub`）采用“平台管理员管理多个 user，每个 user 挂多把 key”的模型。但这是**平台管理员视角**，和 sub2api 企业客户的**自助视角**不同：在 sub2api 中，企业客户本身已经是一个 `users` 行，它的“员工”更自然地映射为该账号名下的多把 `api_keys`，而不是新的登录主体。

这里要做的取舍是：引入子账号 / 组织成员（org-member）这样的新登录实体，还是直接增强现有 `api_keys` 来承载“员工席位”。

## 取舍

**用 `api_keys` 承载企业成员管理，不引入子账号 / 组织成员登录实体。** 一把 Key 等于一个员工席位。

企业管理员通过批量创建、批量限额管理、结构化标签和聚合监控视图来管理旗下所有 Key，而不是管理一组可独立登录的子账号。

## 理由

1. sub2api 数据模型已是 `User 1:N APIKey`（`backend/ent/schema/user.go` edge `api_keys`），Key 天然适合承载员工席位。
2. `api_keys` 已具备配额、5h/1d/7d 限流、过期、分组、IP 黑白名单等控制维度，覆盖企业管理员"分别控制和限制员工用量"的核心需求。
3. 企业管理员的需求是“统一控制和观察”，不是“员工各自独立登录”。Key 方案足以覆盖。
4. 不新增登录主体，避免对认证、计费、权限体系的大规模改动，符合 dev-zz "改动可隔离、与上游冲突面小"的纪律。

## 后果

- 第一阶段先增强 Key 发放能力：新增用户侧批量创建端点，仍使用现有 `api_keys` 字段，不做 schema 迁移。
- 员工只有 Key、没有站点账号时，通过公共只读 Key 状态查询接口查看本人 Key 的余额用量、过期时间、最近使用情况和限流配置；该接口不引入登录主体，不暴露 owner 账号余额或其它 Key。
- 第二阶段再增强 `api_keys` 的结构化分类能力：新增 `tags` 字段、筛选与批量管理能力。
- 企业成员 Key 的 `quota` 表示单把 Key 的用量上限，不表示创建时冻结或预占企业主账号余额。真实扣费仍沿用现有调用时计费链路，从 owner 的余额 / 订阅 / Key quota 体系中结算。
- 改造集中在增强 `api_keys` 与新增批量端点 / owner 视角统计端点，不触碰认证与计费主链路。
- 员工仍没有独立站点账号，但可通过公共只读 Key 状态查询查看本人 Key 的状态、余额用量、过期时间、最近使用时间和限流配置；该接口不暴露 owner 账号余额、其它 Key 或运营成本。
- owner 可以在用户侧 `使用记录` 的 `分析` tab 查看自己名下 Key 的汇总、排行榜、模型、分组、标签和趋势。
- `tags` 字段为 schema 变更，需新增 ent 字段 + migration（jsonb + GIN 索引），并在 `patches.md` 标注迁移编号供合并上游时核对。
- 实施细节见 [企业客户 Key 成员管理](../features/enterprise-key-member-management.md)。

## 替代方案

### 引入子账号 / 组织成员登录实体

新增 org-member 实体，每个员工可独立登录查看本人用量，类似 cch 的多 user 模型。

- 优点：员工自助，颗粒度最细。
- 缺点：需改动认证、计费、权限体系，与上游冲突面大，违背 dev-zz 隔离纪律。仅当企业客户明确要求员工独立登录时才评估，不在本期范围。

### 照搬 claude-code-hub 的数据模型

直接移植 cch 的 user/key 表结构与 Server Actions。

- 缺点：技术栈完全不同（TS/Next.js/Drizzle vs Go/Vue/ent），且 cch 的 user 是平台管理员视角的成员，与 sub2api 企业客户自助视角错位。只借鉴功能形态，不照搬实现。
