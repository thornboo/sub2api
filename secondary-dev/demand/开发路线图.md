## 三、技术实现细节

### 3.1 通用实现模式（参考 Anthropic）

每个平台的实现应遵循以下模式：

```go
// 1. 主入口方法 - 检查是否启用密钥池
func (s *Service) forwardPlatformPassthrough(...) {
    if account != nil && account.IsPlatformAPIKeyPassthroughEnabled() {
        if account.HasAccountAPIKeyPool() {
            // 使用密钥池路径
            return s.forwardPlatformPassthroughWithInput(...)
        }
        // 使用遗留单密钥路径
        return s.forwardPlatformPassthroughWithToken(..., account.APIKey, nil)
    }
    // 其他路径（OAuth 等）
}

// 2. 密钥池调度方法
func (s *Service) forwardPlatformPassthroughWithInput(...) {
    selections := account.EffectiveAPIKeySelectionsForRequest(...)

    var lastErr error
    for _, key := range selections {
        result, err := s.forwardPlatformPassthroughWithToken(..., key.APIKey, &key)
        if err == nil {
            s.markAccountAPIKeyUsed(ctx, key.ID, false)
            return result, nil
        }

        // 检查是否应该故障转移
        var failoverErr *UpstreamFailoverError
        if errors.As(err, &failoverErr) {
            s.cooldownAccountAPIKeyModel(ctx, key.ID, upstreamModel, failoverErr.StatusCode, failoverErr.ResponseBody)
            s.markAccountAPIKeyUsed(ctx, key.ID, true)
            lastErr = err
            continue // 尝试下一个密钥
        }

        // 非故障转移错误，直接返回
        return nil, err
    }

    return nil, lastErr
}
```

### 3.2 错误分类指南

每个平台需要实现错误分类器，判断哪些错误应该触发故障转移：

**应该故障转移的错误**:
- 401: 认证失败（密钥无效或过期）
- 429: 速率限制（该密钥/模型已达到限额）
- 529: 服务过载
- 5xx: 服务器内部错误
- 特定的业务限制 400 错误（如模型不可用）

**不应该故障转移的错误**:
- 400: 请求参数错误（客户端问题，换密钥也无法解决）
- 403: 权限错误（除非是特定的业务限制）
- 已经开始流式响应后的错误（无法回滚）

### 3.3 冷却策略

**模型级冷却** (当前实现):
- 仅冷却失败的密钥/模型组合
- 其他模型在同一密钥上仍可调度
- 其他密钥对同一模型仍可调度

**全局冷却** (待讨论):
- 401 错误可能需要全局冷却整个密钥
- 通过设置 `global_cooldown_until` 字段实现

**冷却时长** (待讨论):
- 当前可能是固定时长
- 未来可以根据错误类型分类：
  - 429: 根据 `Retry-After` 头
  - 5xx: 短期冷却（1-5分钟）
  - 401: 长期冷却或禁用

## 五、风险与注意事项

### 5.1 技术风险

1. **错误分类不准确**
   - 风险: 将不应故障转移的错误进行故障转移，导致无效重试
   - 缓解: 仔细分析每个平台的 API 文档，编写详细的错误分类测试

2. **冷却时长不合理**
   - 风险: 冷却时间过短导致频繁重试失败密钥；过长导致可用密钥被浪费
   - 缓解: 从保守的固定时长开始，后续根据监控数据调整

3. **性能影响**
   - 风险: 密钥池查询和调度逻辑增加请求延迟
   - 缓解: 优化数据库查询，添加必要的索引，考虑缓存

### 5.2 运维风险

1. **迁移复杂性**
   - 风险: 从单密钥迁移到密钥池可能导致配置错误
   - 缓解: 保持向后兼容，提供详细的迁移文档和工具

2. **监控盲点**
   - 风险: 密钥级别的故障可能不容易被发现
   - 缓解: 增强日志和监控，在管理界面显示密钥健康状态

### 5.3 业务风险

1. **故障转移延迟**
   - 风险: 多次故障转移导致用户请求延迟增加
   - 缓解: 限制故障转移次数，快速失败策略

2. **成本增加**
   - 风险: 故障转移导致重复请求，增加上游 API 成本
   - 缓解: 准确的错误分类，避免无效重试

## 六、最终实现决策

以下决策已经与用户确认，作为实现的最终标准：

### 6.1 错误处理策略

**决策 1: 401 认证错误处理** ✅
- **选择**: 长时间冷却（可配置，默认1小时），而非永久禁用
- **理由**: 401通常意味着密钥失效或被禁用，但某些上游可能存在临时性401错误
- **实现要点**:
  - 设置 `global_cooldown_until` 字段为当前时间 + 可配置的冷却时长
  - 冷却时长可在全局配置或账号级配置中设置
  - 如果冷却期过后仍然401，管理界面应提示运营者手动处理该密钥

**决策 2: 流式响应中途出错** ✅
- **选择**: 不进行故障转移，直接返回错误
- **理由**: 一旦开始流式响应，客户端已经开始接收数据，无法回滚
- **实现要点**: 保持当前 Anthropic 实现的逻辑，在开始写入响应字节后禁止故障转移

### 6.2 冷却策略

**决策 3: 冷却时长** ✅
- **选择**: 可配置（全局配置或账号级配置）
- **实现要点**:
  - 提供全局默认配置（建议：429错误5分钟，5xx错误1分钟，401错误1小时）
  - 支持账号级覆盖全局配置
  - 支持根据 `Retry-After` 响应头动态调整冷却时长
  - 配置项示例：
    ```json
    {
      "cooldown_durations": {
        "401": 3600,  // 1小时
        "429": 300,   // 5分钟
        "5xx": 60     // 1分钟
      }
    }
    ```

**决策 4: 手动清除冷却** ✅
- **选择**: 仅管理员可用，需要二次确认
- **实现要点**:
  - 在管理界面提供"清除冷却"按钮
  - 点击后弹出确认对话框，说明风险
  - 记录操作日志（谁在什么时间清除了哪个密钥的冷却）

### 6.3 负载均衡策略

**决策 5: 同优先级密钥选择** ✅
- **选择**: 最近最少使用（LRU，当前实现）
- **实现要点**: 保持当前 `EffectiveAPIKeySelectionsForRequest()` 的实现逻辑

**决策 6: 优先级影响强度** ✅
- **选择**: 严格优先级（高优先级密钥可用时，低优先级永远不用）
- **理由**:
  - 实现简单，行为可预测
  - 运营者可以通过设置相同优先级来实现负载均衡（同优先级内LRU轮询）
  - 软优先级实现复杂，难以调试和预测
- **实现要点**:
